# 05. Run Rapid Update Cycle (WRFDA)

In this step, the **WRF Data Assimilation (WRFDA)** system is used to assimilate weather radar observations into the WRF background fields.

During the **12-hour spin-up period**, radar observations are assimilated every **3 hours** (00, 03, 06, and 09 UTC). Each assimilation cycle updates the model initial condition before launching the next 3-hour forecast.

Three radar sites are used in this training:

- **C-Band Radar Jakarta**
- **X-Band Radar Cengkareng**
- **X-Band Radar Purwakarta**

The following radar observations are assimilated:

- **Reflectivity (dBZ)**, to improve the analysis of precipitation and hydrometeor distributions.
- **Radial Velocity (Vr)**, to improve the low-level wind field and storm circulation.

The updated analysis produced by WRFDA becomes the initial condition for the next forecast cycle. This cycling procedure allows the model to continuously adjust the atmospheric state using the latest radar observations before the free forecast is started.

The overall cycling strategy is illustrated below.

```text
IFS HRES (00 UTC)
        │
        ▼
real.exe
        │
        ▼
Background (wrfinput)
        │
        ▼
Radar Assimilation (00 UTC)
        │
        ▼
Updated wrfinput
        │
        ▼
WRF Forecast (+3 h)
        │
        ▼
Radar Assimilation (03 UTC)
        │
        ▼
Updated wrfinput
        │
        ▼
WRF Forecast (+3 h)
        │
        ▼
Radar Assimilation (06 UTC)
        │
        ▼
WRF Forecast (+3 h)
        │
        ▼
Radar Assimilation (09 UTC)
        │
        ▼
Updated Initial Condition
        │
        ▼
Free Forecast
```

## Step 2. Real.exe
The **real.exe** program converts the meteorological fields generated by WPS (`met_em*`) into the initial and lateral boundary conditions required by WRF.
Copy the provided job real.sh script

```bash
cp -rv /scratch/dmu/training_data/script/real.sh .
```

The real.sh script should contain the following configuration.

```bash
#!/bin/bash
#SBATCH --job-name=real.exe
#SBATCH --partition=riset
#SBATCH --nodes=1
#SBATCH --ntasks-per-node=20
#SBATCH --time=03:00:00
#SBATCH --mem=720GB
#SBATCH --output=real_%j.out
#SBATCH --error=real_%j.err

source ~/.bashrc_wrf

module purge
ml restore wrf_module

source /software/compilers/intel-oneapi/2025.2.1/setvars.sh --force

time mpirun ./real.exe

```

Submit the real.exe job

```bash
sbatch real.sh
```

Monitor the queue 

```bash
squeue -u $USER
```

Monitor the execution log.

```bash
tail -f rsl.out.0000
```

The expected outputs are:

- `wrfinput_d01`
- `wrfinput_d02`
- `wrfinput_d03`
- `wrfbdy_d01`

These files provide the background (first guess) fields that will be updated during the Rapid Update Cycle using radar observations.

## Step 9. Run the Next Forecast Cycle
Before running **wrf.exe**, update the simulation period in **namelist.input** according to the current cycling time.

For example

| Cycle | Forecast Period |
|--------|-----------------|
| 00 UTC | 00–03 UTC |
| 03 UTC | 03–06 UTC |
| 06 UTC | 06–09 UTC |
| 09 UTC | 09–12 UTC |

After updating the namelist, run

```bash
wrf.exe
```

The output forecast provides the background field for the next radar assimilation cycle.

Repeat the cycling procedure every **3 hours** until the end of the **12-hour spin-up period**.

The final updated initial condition produced at **12 UTC** will be used as the starting point for the **free forecast** described in the next module.
